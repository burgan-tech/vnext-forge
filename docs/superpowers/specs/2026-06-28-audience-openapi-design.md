# Audience-Based OpenAPI Export

**Date:** 2026-06-28  
**Status:** Approved

## Context

The existing "Preview OpenAPI" feature generates a full OpenAPI 3.1 spec covering all externally-triggerable transitions in a workflow. This is a complete picture, but integrators often need a **role-scoped view** â€” e.g., "what endpoints can the `manager` role call?"

Additionally the current generator ignores workflow-level `_comment` fields and uses only English for label resolution. This spec addresses all three gaps:

1. **Audience filter** â€” role-scoped OpenAPI output with a new toolbar entry point
2. **`_comment` propagation** â€” workflow and transition comments appear in `info.description` / operation `description`
3. **Language-aware labels** â€” `summary` and `title` fields use a user-selected language with first-available fallback

## 1. Generator Layer (`packages/doc-gen`)

### 1.1 `openapi-doc.ts` changes

**`firstLabel` signature update:**

```ts
function firstLabel(
  labels: { language?: string; label?: string }[] | undefined,
  fallback: string,
  language = 'en'
): string
```

Prefers the label whose `language` matches `language` (exact or prefix match, e.g. `"tr"` matches `"tr-TR"`). Falls back to the first available label, then to `fallback`. Existing call sites that omit `language` continue to default to `'en'`.

**Extended internal types:**

```ts
interface RoleGrant { role: string; grant: 'allow' | 'deny' }

interface TransitionLike {
  // ...existing fields...
  roles?: RoleGrant[];
}

interface WorkflowLike {
  _comment?: string;
  // ...existing fields...
}
```

**New options parameter:**

```ts
export interface OpenApiOptions {
  /** When set, only transitions visible to at least one of these roles are emitted. */
  audienceRoles?: string[];
  /** Preferred language code for label/summary resolution (e.g. 'en', 'tr'). Default: 'en'. */
  language?: string;
}

export function buildWorkflowOpenApi(
  workflowJson: unknown,
  resolvers: OpenApiResolvers,
  options?: OpenApiOptions
): OpenApiDocument
```

**Audience filter function:**

```ts
function isVisibleToAudience(
  roles: RoleGrant[] | undefined,
  audienceRoles: string[]
): boolean {
  if (!audienceRoles.length) return true;        // no filter active
  if (!roles?.length) return true;               // no restrictions â†’ open to all
  for (const sel of audienceRoles) {
    if (roles.some(r => r.role === sel && r.grant === 'deny')) return false; // DENY wins
  }
  return audienceRoles.some(sel => roles.some(r => r.role === sel && r.grant === 'allow'));
  // No matching allow â†’ default DENY â†’ false
}
```

Applied to: `collectExternalTransitions` output (regular, shared, cancel, exit, updateData) before emitting paths. `startTransition` is **always included** â€” it represents workflow initiation, not role-gated state progression.

**`_comment` in `info.description`:**

```ts
const wfComment = (workflowJson as WorkflowLike)._comment;
info.description = [
  `OpenAPI specification for the "${wfKey}" workflow (domain "${domain}"). Generated by vNext Forge.`,
  wfComment ? `\n\n${wfComment}` : '',
].join('');
```

Transition `description` already uses `t._comment`; this is preserved.

**Audience context in `info.description`:**

When `audienceRoles` is non-empty, append:
```
\n\n**Audience:** admin, manager
```

### 1.2 New export `collectWorkflowRoles`

New file `packages/doc-gen/src/generators/openapi-audience.ts` (exported from package index):

```ts
export function collectWorkflowRoles(workflowJson: unknown): string[]
```

Scans all role grants from:
- `attributes.startTransition.roles`
- `attributes.states[].transitions[].roles`
- `attributes.sharedTransitions[].roles`
- `attributes.cancel?.roles`, `attributes.exit?.roles`, `attributes.updateData?.roles`
- `attributes.states[].queryRoles`

Returns unique role names sorted alphabetically.

Also exports:

```ts
export function collectWorkflowLanguages(workflowJson: unknown): string[]
```

Scans all `labels[].language` values across the workflow. Returns unique language codes sorted, e.g. `['en', 'tr']`. Used to populate the language dropdown in the picker dialog.

## 2. UI Layer (`packages/designer-ui`)

### 2.1 New: `AudienceRolePickerDialog.tsx`

**Props:**

```ts
  open: boolean;
  onOpenChange: (open: boolean) => void;
  workflowJson: unknown;
  onGenerate: (filter: { roles: string[]; language: string }) => void;

```

**Behaviour:**
- On `open`: calls `collectWorkflowRoles(workflowJson)` and `collectWorkflowLanguages(workflowJson)`
- Shows a multi-select checklist of role names (all unchecked by default)
- Shows a language dropdown populated from `collectWorkflowLanguages`; defaults to `'en'` if available, otherwise first available

- On submit: calls `onGenerate({ roles: selectedRoles, language })` and closes

**Edge case:** If the workflow has no roles defined anywhere, show an info banner â€” "No roles defined in this workflow. The generated spec will include all transitions." The dialog still allows language selection and proceeds normally.

## 3. Filter Semantics Summary

| Transition `roles` | Selected audience roles | Result |
|---|---|---|
| `undefined` or `[]` | any | âś… Always included (open to all) |
| has roles | `[]` (no selection) | âś… Included (no filter active) |
| has roles | selected role has `deny` | âťŚ Excluded (DENY wins) |
| has roles | selected role has `allow`, no deny | âś… Included |
| has roles | selected roles not in transition roles | âťŚ Excluded (default DENY) |
| `startTransition` | any | âś… Always included |

## 4. `_comment` and Labels Mapping

| Workflow field | OpenAPI output field |
|---|---|
| Root `_comment` | `info.description` (appended after base text) |
| `attributes.labels` | `info.title` (language-aware `firstLabel`) |
| Transition `_comment` | Operation `description` |
| Transition `labels` | Operation `summary` (language-aware `firstLabel`) |
| State labels | Not surfaced (states don't map to operations) |

Markdown is supported natively in OpenAPI 3.1 description fields (CommonMark) and Redoc renders it â€” `_comment` values are included verbatim, no escaping needed.
